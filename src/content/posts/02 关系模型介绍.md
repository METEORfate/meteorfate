## 关系模型概述<sup>`§2.0`</sup>
关系模型由关系数据库、关系操作集合和关系完整性约束三部分组成
+ 单一的数据结构——`关系`(`Relation`)
	+ 实体和实体之间的联系都用关系描述
+ 关系操作
	+ 特点：一次一集合
	+ 语言：关系代数、ALPHA、元组关系演算、QBE、SQL…
+ 三类完整性约束
	+ 实体完整性
	+ 参照完整性
	+ 用户定义完整性

### 概念总结
+ `关系`(`Relation`)：对应不可分的二维表
+ `元组`(`Tuple`)：表中的一行即为一个元组
+ `属性`(`Attribute`)：表中的一列即为一个属性
+ `码`(`Key`)：表中的某属性组，它可以唯一确定一个元组
+ `域`(`Domain`)：属性的取值范围
+ `分量`：元组中的一个属性值（联系矢量概念）
+ `关系模式`(型)：对关系的描述
+ `关系名（属性１，属性２，……，属性n）`

## 关系模式(Relation Schema)<sup>`§2.2`</sup>
<!--{"show":{"startStep":1}}-->
### 关系模型-形式化定义(Formal Definition)<sup>`§2.2.0`</sup>
#### 域（Domain）
域是具有相同数据类型的值的集合
+ `笛卡儿积`(卡氏积 `Cartesian Product`)  
	给定一组域D1,D2,…,Dn,这些域中可以有相同的。D1,D2,…,Dn的笛卡儿积为：
	$$D1 \times D2 \times … \times Dn＝\{ (d1,d2,…,dn)| di \in Di, i=1,2,…n \}$$
	其中每一个元素称为一个`n元组`（`n-tuple`），元素中的每个值di称为一个`分量`
+ 例：  
	D1={大，中，小}，D2={红，绿}  
	D1×D2={(大，红)，(大，绿)，(中，红)，(中，绿)，(小，红)，(小，绿)}  
	D2×D1={(红，大)，(绿，大)，(红，中)，(绿，中)，(红，小)，(绿，小)}  
```notice 
笛卡儿积不满足交换律，D1×D2≠D2×D1；即笛卡儿积中元素（元组）的各分量是有序的。
```
+ `基数`：  
若$Di(i=1,2,…,n)$为有限集，其基数为$|Di|$，则$D1×D2×… ×Dn$的基数为：$|D1|×|D2|×… ×|Dn|$
#### 关系（Relation）
+ 定义
	+ $D1 \times D2 \times … \times Dn$ 的**任意子集**叫做在域D1,D2,… ,Dn上的关系，可记做：`R(D1,D2,… ,Dn)`，R为关系名，n是关系的`目`或`度`(`degree`)
		```notice
		注意: 这里的“子集”是“任意子集”，包括空集
		```
	+ `单元关系`(`Unary relation`)和`二元关系`(`Binary relation`)：目n=1和n=2的关系
	+ 笛卡儿积不满足交换律，而关系通过给关系的列附加属性名的方式取消元组各分量的有序性
	+ 关系可以是无限集。通常在关系数据模型中限定关系为有限关系
+ **`属性`**(**`Attribute`**)  
	构造笛卡尔积的域可以重复，意味着关系中的一个元素(元组)的两个不同分量可能来自同一个域，无法通过其来自的域区分是哪个分量。为了区分，为元组的分量(列)取<z md-most>不同的名字</z>，即属性。n目关系有n个属性。
+ 基本关系六性质(三列两行一分量)
	+ 列是同质的(来自同一个域)
	+ 不同列可出自同一个域，每一列为一个属性，不同属性(列)给不同属性名
	+ 列的顺序可任意交换
	+ 任意两个元组不能完全相同
	+ 行(元组)的顺序可任意交换，即：关系是无序(`unordered`)的
	+ 每一分量是不可分的数据项
#### 属性的类型<sup>`§2.1`</sup>
+ 某个属性所有允许值的集合称为属性的域(`domain` of the attribute)
+ 属性的值一般要求是“`原子的`”(`atomic`)，即不可分割
+ 有个特别的值：`null`  ，所有的域中都有这个成员(参见:[什是空值(null)](#null))
	+ null 值会导致许多操作定义复杂化
### 关系模式/Relation Schema
关系模式是对关系的描述(是对型的描述)。需描述元组集合的结构：由哪些属性构成，这些属性来自哪些域，以及属性与域之间的映象关系
+ 关系模式的符号表述：$R(U，D，dom，F)$  
	$R$为`关系名`、$U$为组成该关系的`属性名集合`、$D$为属性组U中属性所来自的`域的集合`、$dom$为属性向域的`映象集合`、$F$为属性间数据的`依赖关系集`
+ 通常可以简记为：$R(U)$ 或 $R(A1,A2, …,An)$  
	其中R为关系名，A1,A2, …,An为属性名。而域名及属性向域的映象常常直接说明为属性的类型、长度
<!--{"show":{"css":"font-size:1.5em"}}-->
### 模式(schema)与实例(instance) - 关系/Relation
+ 关系是关系模式的`实例`(`instance`)，是关系模式在某一时刻的状态或者内容
	+ 关系是同型元组的无序集合
	+ 关系r的一个成员t是一个元组(tuple),对应简单二维表里的一行
+ 关系模式是静态稳定的，而关系是动态时变的
+ 关系模式定义了元组的语义
	+ 一个n目谓词
	+ 使该n目谓词为真的笛卡尔积中的元素的全体构成该关系模式中的关系
## 码(Key)<sup>`§2.3`</sup>
有关系模式: $R(U)$
+ `超码`(`superkey`)
	令$K \subseteq U$ ，如果确定了$K$中各属性的值就能够在$R(U)$的每个可能关系$r$中标记出的唯一元组(或确定$r$中无此元组)，则$K$是$R$的超码	
+ `候选码`(`Candidate key`)  
	如$K$是$R$的超码，而其**真**子集不是超码，则称属性组$K$为该关系的一个候选码
+ `主码`(`Primary key`)  
	一个关系可能有多个候选码，则选定其中一个作为主码
+ `主属性`(`Prime attribute`)  
	包含在任何候选码中的属性
+ `非主属性`(`Nonprimary attribute`)  
	不包含在任何候选码中的属性，又称：`非码属性`(`None-key attribute`)
+ `全码`(`All-key`)  
	关系模式的所有属性组构成此关系模式的唯一候选码

## 关系模型 - 完整性(Integrity)约束条件<sup>`§2.3`</sup>
+ 实体完整性(Entity Integrity)
+ 参照完整性(Reference Integrity)
+ 用户定义完整性(Custom Integrity)
其中实体完整性和参照完整性是关系模型必须满足的完整性约束条件，被称作是**关系的两个不变性**

### 什么是空值(null)<a id="null"></a>
+ 盘古、上帝、自然数和零
	+ 0:什么都没有
+ NULL
	+ C++中的null和0
	+ 什么都不是和什么都是的null
		+ “说似一物即不中”
### 实体完整性(Entity Integrity)
1. `候选码`是关系中元组的`值标识`，所以同一关系中<b md-most>候选码取值不能重复</b>
2. 若属性$A$是关系模式$R$的<b md-most>主属性</b>，则属性$A$<b md-most>不能取空值</b>
```warning
注意:实体完整性约束规定候选码的所有属性都不能取空值，不仅是候选码整体不能取空值  
```
```tip
工程实践中实体完整性约束常采用“主码中的属性”
```
<!--{"show":{"sub":"--li1-size:1.16;--li2-size;1.08"}}-->
### 参照完整性(Reference Integrity)
<!--{"show":{"sub":"line-height:1.6"}}-->
+ 引用  
	现实世界中的实体之间往往存在某种联系，在关系模型中实体及实体间的联系都是用关系来描述的。这样就自然存在着关系与关系之间的引用
	<!--{"show":{"sub":"line-height:1.3"}}-->
	+ 例1：
		+ 学生（<u>学号</u>，姓名，性别，<i md-most>专业号</i>，年龄）  
		+ 专业（<u>专业号</u>，专业名）
	<!--{"show":{"sub":"line-height:1.3"}}-->
	+ 例2：
		+ 学生（<u>学号</u>，姓名，性别，年龄）
		+ 课程（<u>课程号</u>，课程名）
		+ 选修（<u md-most>学号</u>，<u md-most>课程号</u>，成绩）
+ **`外码`**(**`Foreign Key`**)  
	有关系模式 $R(U)$，令 $F \subseteq U$ ，如果$F$与关系模式$S$的主码$K_S$相对应，则称$F$是$R$的外码，并称$R$为`参照关系`(`Referencing relation`)，$S$为`被参照关系`(`Referenced relation`)或`目标关系`(`Target relation`)。关系$R$和$S$可以是同一个关系
	+ 例：学生（<u>学号</u>，姓名，性别，年龄，<i md-most>同桌学号</i>）
		+ 外码不一定与对应主码同名
		+ R与S不一定是不同的关系
+ **`参照完整性`**
	若$F$是关系模式$R$的<b md-most>外码</b>，被参照关系为$S$，则$R$的实例中每个元组在$F$上的值必须
	1. <b md-most>或者取空值</b>（$F$的每个属性值均为空值）
	2. <b md-most>或者等于</b>$S$实例中某个元组的<b md-most>主码值</b>
<!--{"show":{"css":"font-size:1.4em","sub":"--li1-size:1.15"}}-->
### 用户定义的完整性(User-defined Integrity)<sup>`§2.3.0`</sup>
+ 实体完整性和参照性适用于任何关系数据库系统。除此之外，不同的关系数据库系统<x-x md-most>根据其应用环境的不同</x-x>，往往还需要一些<x-x md-most>特殊的约束条件</x-x>。
+ <x-x md-most>用户定义的完整性</x-x>就是针对某一具体关系数据库的约束条件，它反映某一具体应用所涉及的数据必须满足的语义要求。
+ 关系模型应提供定义和检验这类完整性的机制，以便用统一的系统的方法处理它们，而不要由应用程序承担这一功能

一个数据库包含若干关系，<z md-most>数据库模式</z>就是这些<z md-most>关系的模式的集合</z>  
如学校管理数据库中包含信息：instructor 、student、advisor……
+ 糟糕的设计: <br><code style="font-size:80%">univ (instructor-ID, name, dept_name, salary, student_Id, ..)</code><br>导致
	+ 重复信息（例如，两个学生有同一个教师）
	+ 需要空值（例如，表示没有顾问的学生）

## 关系模型-数据操作/存储模式/优劣<sup>`§2.0`</sup>
+ Operations
	+ 一次一集合方式：只需指出“What to do”，而非关模型中需要告诉系统“How to do”。
+ Storage Schema(Physical Level Schema)
	+ 表代表所有实体及其联系
	+ 物理存储方式极其灵活
+ Good & Bad
	+ 严格的数学基础
	+ 概念单一，结构简单清晰
	+ 存取路径对用户透明
